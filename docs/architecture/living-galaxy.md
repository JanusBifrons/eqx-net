# Living Galaxy — design & roadmap

> **Status: design / roadmap (no code yet).** This document is the review vehicle
> for a multi-PR effort. Each "Workstream" below is implemented as one or more of
> the phases in [Phased roadmap](#phased-roadmap); nothing here has shipped. Land
> the phases in order, each with its own CLAUDE.md / `docs/` updates per the repo's
> phase-gate ritual.

## Context

EQX Peri today has a working but *thin* galaxy: a 7-sector "sunflower" (one centre
+ six leaves) hard-coded in [src/core/galaxy/galaxy.ts](../../src/core/galaxy/galaxy.ts),
a Pixi [`GalaxyMapLayer`](../../src/client/render/galaxy/GalaxyMapLayer.ts) that
draws bare green hexes, a clean inter-sector warp/transit flow, and a
[`LivingWorldDirector`](../../src/server/livingworld/LivingWorldDirector.ts) that
roams 24 hunter bots.

A sibling project (`eqx-peri`, a separate, older, single-player codebase) has a
much richer galaxy *vision*: subtly highlighted faction territories, per-sector
icons, a procedurally-shaped map with regions, and a Sector-Conquest mode. This
document plans how to bring that **vision** (not the code) into this multiplayer
architecture, across four threads:

1. A **living** galaxy map with the eqx-peri visual polish brought across in full:
   faction/ownership **territory tint**, the **contiguous-territory hover-shrink**
   animation (a whole same-faction region scales toward its centroid as one unit —
   *not* a per-hex effect), and a **rich per-sector icon set** (structures, live
   enemy/neutral ship counts, plus environmental feature glyphs) — all updating
   while you watch. **The shrink + icons are core requirements, not optional.**
2. The map shown **front-and-centre on page load** as the first screen; clicking a
   sector routes to login (logged out) or ship-select (logged in), collapsing
   today's separate "Join the fight!" landing screen into the map.
3. A **bigger, more interesting galaxy** — grow 7 → ~20-25 sectors with a
   multi-region/chokepoint shape instead of the sunflower, generated by a method
   adapted from eqx-peri.
4. The roaming-group population **scaled up proportionally** so the larger galaxy
   feels alive without exploding drone counts or the server tick budget.

## Scope & locked decisions

- **Faction control:** the per-sector data model is *capable* of real ownership +
  faction control, but v1 renders **cosmetic/static** faction assignment, with
  explicit comments/notes for future expansion (derived ownership → named factions
  → conquest — see [Forward-compatibility](#forward-compatibility)).
- **Live data:** a new `GET /galaxy/snapshot` REST endpoint; the client polls every
  3-5 s. Not sub-second — "alive" via a few-second refresh.
- **Drone scaling:** proportional + bounded — grow squads ~linearly with sectors
  (3 → ~7-8 squads, ≈56-64 bots), keep the director's concentrate-on-players logic
  so an idle galaxy stays cheap.

**Why eqx-peri's generator can't be ported directly:** eqx-peri generates 90
**cosmetic hex cells** via frontier-expansion + BFS faction flood-fill, regenerated
every time the map opens. EQX Peri sectors are **heavyweight Colyseus rooms** (one
room per sector), capped at ~20-25, hand-authored + test-validated for multi-VM
determinism. The unit differs (cosmetic cell vs. live room), so we adapt the
*ideas* — seeded organic growth, regions, contiguous territory, edge adjacency —
into a **node-graph generator whose frozen output is baked into `galaxy.ts`**
(preserving the pure-module + symmetric-edge-test invariant).

## Workstream A — Galaxy expansion (7 → ~20-25, chokepoint shape)

### Approach: generate → bake → validate

Write a **deterministic, seeded** generator (`scripts/generate-galaxy.ts`, run via
`tsx`) that emits a `GALAXY_SECTORS` literal; hand-review and paste the output into
`src/core/galaxy/galaxy.ts`. `galaxy.ts` stays a pure hand-authored-equivalent
constant locked by `galaxy.test.ts`. The generator is a reproducible dev tool (a
fixed seed → the same galaxy), so reshuffles are a re-run + paste, never runtime
RNG (which would break the determinism / multi-VM invariant).

### Target topology — regions + chokepoints (replaces the sunflower)

Design ~21 sectors (tunable 20-25) as **a small home core + 3 frontier regions,
each reachable only through a chokepoint gateway**:

- **Core (home) region** — `sol-prime` (kept as centre) + 2 adjacent core sectors.
  Dense asteroids, lowest drone presence, the safe hub players spawn into.
- **3 frontier regions** (~5 sectors each), themed for future faction assignment
  (nodding to eqx-peri's empires: e.g. *Verdant Reach / Crimson Expanse / Azure
  Deep*). Sparser cover, higher drone traffic toward the edges.
- **Chokepoint sectors** — each frontier region connects to the core through
  exactly **one** gateway sector, forcing routing through bottlenecks. Add 1-2
  cross-links between adjacent frontier regions (also via a shared chokepoint) for
  interesting back-routes, not a fully-connected mesh.
- **Entry (edge) sectors** = the frontier-most sector(s) of each region — where
  drone squads warp in and roam inward toward the chokepoints/core.

This gives the wave/transit gameplay real geography (defend a chokepoint; a wave
must funnel through it) and a natural home for cosmetic faction territories now +
real contested ownership later.

### Generator algorithm (adapted from eqx-peri)

Operate in **axial-hex space** (the map + landing screen project via `axialToPixel`,
so coords must be valid hex positions, collision-free):

1. Place each region's **seed hex** at a chosen axial position spread around the
   core (seeded jitter for organic placement, like eqx-peri's frontier pick).
2. Grow each region by seeded frontier-expansion to its target size, keeping cells
   contiguous (eqx-peri's neighbour-count gating).
3. Insert a **chokepoint hex** on the core↔region path; wire symmetric edges so the
   region attaches to the core *only* through it.
4. Assign per-sector metadata by region theme: `name` (from a curated pool +
   hand-polish), `description`, `asteroidConfigKey`, `droneCount: 0`
   (`AMBIENT_DRONE_FLOOR`), `defaultSpawn: {0,0}`.
5. Emit symmetric `neighbours`, the baked **entry-sector set**, a static cosmetic
   **`faction` tag per region**, and a static **`features` list per sector**
   (eqx-peri-style flavour glyphs — see Workstream B).

**Region = faction = contiguous territory (load-bearing for the hover-shrink).**
Every sector in a frontier region shares one `faction` tag, and the region's
sectors are contiguous in the graph by construction. That contiguity is exactly
what makes eqx-peri's territory-shrink meaningful: hovering any sector in a region
collects its whole same-faction territory (BFS over same-faction neighbours) and
shrinks them together. The core's home sectors are their own (neutral-ish) faction
or singletons. The generator must therefore keep each faction's sectors
**graph-contiguous** — a hard generator invariant, asserted in `galaxy.test.ts`.

### Required code changes

- **Keep the existing 7 keys** (`sol-prime`, `orion-belt`, …) and *add* ~13-18 new
  ones, re-wiring neighbours/positions into the new shape. Keys are persistence
  identities — persisted snapshots (`game_snapshots.sector_id`) and roster
  `last_sector_key` reference them; reusing keys preserves continuity. Adjacency is
  re-derived at runtime (not persisted), so re-wiring is safe.
- **Entry sectors must become explicit, not max-hex-distance-derived.** With a
  multi-region shape the regions sit at *different* hex distances, so the current
  `getEntrySectors()` max-distance rule would pick only the single farthest region's
  edge — wrong. Change `galaxy.ts` to read a **baked `ENTRY_SECTOR_KEYS` set** (the
  generator marks each region's frontier sectors); keep the `getEntrySectors()` /
  `isEntrySector()` signatures identical so `population.ts` and the director are
  untouched. Add a `galaxy.test.ts` case: every entry key resolves, ≥1 per region,
  never the core.
- `galaxy.test.ts`: regenerate expectations — N sectors, symmetric edges, no
  dangling neighbours, connected graph, chokepoints have the intended degree, entry
  set valid, faction sets contiguous. (The "exactly 6 ring outers at distance 1"
  assertion is replaced.)
- **No `CURRENT_SCHEMA_VERSION` bump** — adding sectors doesn't change snapshot
  *shape*; new sectors simply have no prior row → fresh spawn.
- `GalaxyRegistry.resolveSectorConfig` + `index.ts` eager-create loop scale
  automatically off `GALAXY_SECTORS` (verify boot prints N `galaxy room created`).

### Server cost of more rooms

25 eager rooms vs 7. Each empty galaxy room runs only the sim step (broadcast
short-circuits at 0 clients) and — with drone-warp-in (`droneCount: 0`) — holds **no
entities** until a squad roams in, so an empty room's per-tick cost is near-zero.
Boot adds ~2-3 s of room creation. The real population cost is the director
(Workstream D), not the room count.

## Workstream B — Living map (faction tint + icons + live counts)

### Per-sector live-state data model (forward-compatible)

Define a shared shape (in `src/shared-types/`, zod-validated) returned by the new
endpoint and consumed by the map:

```ts
interface SectorLiveState {
  key: string;
  players: number;        // active hulls in the room
  enemies: number;        // hostile drones (kind 1) in interest/room
  neutrals: number;       // neutral roaming drones (kind 1, not hostile)
  structures: number;     // placed structures (StructureRegistry count)
  // Ownership/faction — CAPABLE of real ownership; v1 is cosmetic/static.
  owner: {
    factionId: string;    // v1: the static region faction from galaxy.ts
    contested: boolean;   // v1: always false; FUTURE: >1 base owner present
  } | null;               // null = unclaimed/neutral
}
```

**Cosmetic/static for now, with explicit TODOs:** `owner.factionId` is the static
per-region faction baked in `galaxy.ts`; `contested` is hard-`false`. Comment the
seam clearly: *FUTURE — derive `owner` from `StructureRegistry` (who holds the
dominant Capital), set `contested` when multiple owners hold bases; then named
factions; then a conquest flip.* The counts are **already live**; only ownership is
static. (`faction` + `features` reach the client from the bundled static graph, not
this endpoint — see the renderer section.)

### Server — `GET /galaxy/snapshot`

- New handler in `galaxyRouter.ts` returning `{ sectors: SectorLiveState[] }`.
- Aggregation source: the `LivingWorldDirector` already holds every `galaxy-*` room.
  Add a `galaxySnapshot()` method that, per room, counts swarm entities by kind +
  hostility (`swarm.all()`; asteroids kind 0 excluded), counts `structureRegistry`,
  reads `playerCount()`, and stamps the static region faction.
- **Cache it on the director's existing ~1.5 s control tick** (compute once, store,
  serve from cache) so HTTP requests are O(1) and we never recompute per request.
- Wire the router → director via an injected accessor singleton
  (`getGalaxyStatsProvider()`, set in `index.ts` beside the director — the
  `incomingPlayerSink.ts` pattern), **null-guarded**: when the director is disabled
  (`EQX_DISABLE_LIVING_WORLD`) return static-graph + zero counts, never crash.
- Not a live-loop path → **no netgate**, no snapshot-cadence concern.

### Client — poll + thread to the worker-side layer

- `useGalaxyStats` hook polls `/galaxy/snapshot` every 3-5 s, writes a Zustand
  singleton (`galaxyStats: Map<key, SectorLiveState>` — discrete, low-frequency,
  **not spatial** → Zustand-purity-clean). Active only while a galaxy map is on
  screen.
- Cross the worker boundary with a new `SET_GALAXY_STATS { stats }` mainToWorker
  protocol message (mirrors `SET_CURRENT_SECTOR`; plain structured-cloneable data).
  `GalaxyMapLayer.setGalaxyStats(stats)` stores it and repaints.

### Renderer — `GalaxyMapLayer` gets the eqx-peri polish (REQUIRED)

This is the heart of the feature, not a tint pass. Three first-class pieces:

**1. Faction territory tint.** Fill each hex from its `faction` tag → a region
colour. Keep the current-sector pulse + selectable/locked strokes layered on top.
Draw a bolder territory **border** only on the *outer edge* of each contiguous
faction region (suppress the stroke where two same-faction hexes meet) — eqx-peri's
"territory outline, not per-cell grid" look.

**2. Contiguous-territory hover-shrink (essential — the polish that sells it).**
Port eqx-peri's behaviour faithfully:

- On hover (desktop pointer) / press-or-current-sector (touch), collect the hovered
  sector's **contiguous same-faction territory** via BFS over same-faction
  neighbours (port of eqx-peri's `getContiguousTerritory`; a neutral/singleton
  sector is its own territory of one).
- **Shrink the whole territory toward its centroid as one unit** — animate a scale
  from `1.0 → ~0.94` with a per-frame lerp (eqx-peri's `HOVER_SHRINK` / `HOVER_LERP`)
  in the existing `tick`. The territory scales together, not each hex independently.
- **Smooth transitions:** moving to a new territory animates the outgoing one back
  to `1.0` while the incoming one shrinks in (eqx-peri's incoming/outgoing slot
  containers); moving *within* the same territory only redraws the root-hex
  highlight (cheap).
- **Eraser layer:** an invisible background-hex layer beneath the scaled territory
  hides the static borders the scale would otherwise expose (eqx-peri's technique).

This requires a **render-tree refactor of `GalaxyMapLayer`**: today hexes are flat
children of one `hexLayer`. Group them into **per-territory sub-containers**
(positioned at the territory centroid, children offset relative to it) so a single
`container.scale.set(s)` shrinks the whole territory toward its centroid. Build the
territory grouping once from the static graph (faction-contiguous BFS); rebuild only
if the graph changes. The pure grouping + centroid + BFS logic lives in a
unit-tested helper (`galaxyTerritories.ts`, the `galaxyLayerDecisions.ts` idiom) so
the Pixi layer stays thin. Touch handling: the layer already routes pointer events
(`onPointerMove`/`onPointerUp`); desktop hover drives the shrink, touch uses the
pressed/selected territory (no true hover) — documented, not faked.

**3. Rich per-sector icons (REQUIRED — both live + flavour).** Small oversampled
Pixi `Graphics`+`Text` glyphs near each hex (`MAP_LABEL_RESOLUTION`, built once per
entry, updated in `repaint()`):

- **Live gameplay icons** (from `/galaxy/snapshot`): a structure glyph + count when
  `structures>0`; an enemy-ship count; a neutral-ship count; a player-presence dot.
- **Environmental feature glyphs** (from the static baked `features` per sector —
  the eqx-peri icon vocabulary): asteroid-rich, nebula, rich-minerals, black-hole,
  station/NPC. `features` is a static cosmetic list on `GalaxySector`
  (`SectorFeature = 'asteroid' | 'nebula' | 'minerals' | 'blackhole' | 'station'`),
  baked by the generator from region theme + `asteroidConfigKey`. This is what gives
  each sector character on the map, exactly like eqx-peri.
- Per the "start tiny" rule, glyphs are small; in `overlay` mode a zoom/declutter
  threshold may hide counts when the cluster is tiny (noted, tunable).

`faction` + `features` reach the client from the **bundled static graph**
(`GALAXY_SECTORS`), so no wire cost; only the live counts ride `/galaxy/snapshot`
and the worker `SET_GALAXY_STATS` message.

**Both modes** get all three: the in-game `overlay` shows live intel + territory
colour about neighbours (great for warp decisions) while staying transparent; the
full-screen `selector` is the living landing map with the full hover-shrink polish.

`load → galaxy map (selector, live) → tap sector → [login if logged out] →
ship-select → game`. Collapse the standalone "Join the fight!" screen into the map.

## Workstream D — Director / roaming-group scaling (proportional, bounded)

### Change

`LIVING_WORLD_SQUAD_COUNT` 3 → **~7** (keep `SQUAD_SIZE = 8`) → **~56 bots** (tune
7-8 squads / 56-64 bots). Squads home spread across the (now multi-region) entry
sectors at boot. The wave/roam/dispatch machinery is **size-invariant** (BFS over
the graph) and needs no structural change — only the count moves. The chokepoint
topology makes roaming squads funnel through gateways, creating organic contested
zones.

### Per-sector cost math (for sign-off)

- **Idle galaxy (no players):** 56 bots spread across ~25 sectors ≈ **2-3 active
  drones/sector**, but squads stay *clustered in formation* (not smeared), so most
  sectors hold 0 and a few hold a full squad. Empty rooms cost ~0 (no entities).
  Drone-sim cost scales ~linearly with *total live bots* (≈2.3× the current 24) but
  is spread across independent room ticks; each room stays far under its 16.67 ms
  budget. The measured drone-mount tick was ≈0.016 ms avg / 7 sectors — headroom is
  ample.
- **Player present:** unchanged feel — a wave is still ~1 squad (8 bots) per ready
  base; `dispatchIntervalMs` (5 min/faction) and "nearest squad" dispatch are
  untouched. More squads just means more *simultaneous* waves across more player
  bases (scales with player count, which is the point).
- **Roaming coverage:** 25 sectors × 45 s/hop is a long cycle; more squads improve
  coverage so the bigger galaxy doesn't feel empty.

### Required

- Bump the constant in `director/SquadPool.ts`; verify `population.ts`
  apportion/entry-pick handle more sectors (size-invariant — confirm with a test).
- **Netgate applies** (root CLAUDE.md invariant #8): the director touches the swarm
  broadcast. Run `pnpm e2e:netgate` (CI) for the scaling change.
- Bench the idle-galaxy server tick (`tick_budget`) at the new bot count to confirm
  the cost math before merge.

## Forward-compatibility

The cosmetic v1 must not box us in. The expansion ladder, in order:

1. **v1 (this roadmap):** static per-region faction tint; counts live; ownership
   model present but `owner.factionId` = static, `contested` = false.
2. **Derived ownership:** compute `owner` from `StructureRegistry` (dominant Capital
   holder); set `contested` when multiple base owners coexist. Reuses the existing
   [`FactionLedger`](../../src/server/faction/FactionLedger.ts) /
   [`Faction.ts`](../../src/core/faction/Faction.ts) system — no new wire.
3. **Named NPC factions:** introduce 2-3 named factions that own/contest territory
   (eqx-peri empires) — a net-new faction-membership model.
4. **Sector conquest:** capture objectives/timers (eqx-peri's Territory Control
   Unit) flip ownership — its own multi-phase effort.

The `SectorLiveState.owner` shape is the stable seam across all four.

## Phased roadmap

Ordered so each phase is independently shippable + testable:

- **Phase 1 — Galaxy expansion.** Generator script + baked `GALAXY_SECTORS` +
  explicit entry set + `faction`/`features` fields + `galaxy.test.ts` rewrite +
  `galaxy-graph.md` refresh (incl. the stale `GalaxyOverviewScreen` fix). Gate:
  graph tests green, server boots N rooms, transit between new neighbours works.
- **Phase 2 — Director scaling.** Squad-count bump + cost bench + netgate. Gate:
  bench under budget, netgate baseline-green, living-world tests green.
- **Phase 3 — `/galaxy/snapshot` endpoint.** Director aggregation + cache + injected
  provider + zod shape + endpoint test. Gate: endpoint returns live counts for all
  sectors, null-guarded under `EQX_DISABLE_LIVING_WORLD`.
- **Phase 4 — Living map render** (the centrepiece; ship in two steps):
  - **4a — territory model + tint + hover-shrink.** Refactor `GalaxyMapLayer` to
    per-territory sub-containers; add the pure `galaxyTerritories.ts` (faction-
    contiguous BFS + centroid); faction tint + outer-edge territory borders;
    contiguous-territory hover-shrink with smooth incoming/outgoing transitions +
    eraser layer. Gate: hovering a region shrinks the whole contiguous territory as
    one unit (E2E via a transform hook on the territory container); unit tests for
    BFS/centroid.
  - **4b — live counts + icons.** `SET_GALAXY_STATS` protocol + `useGalaxyStats`
    poll + live count icons (structures / enemy / neutral / player) + static feature
    glyphs (asteroid/nebula/minerals/blackhole/station). Gate: map shows
    tint/icons/counts in both modes; E2E asserts counts render and refresh ~every
    3-5 s.
- **Phase 5 — Landing-flow merge.** Initial phase → `galaxy-map`, auth-gate on pick,
  fold `MetaLandingScreen` bits, update affected specs + client CLAUDE.md. Gate:
  map-on-load + login-gated-pick + logged-in-pick E2E green; deep-links still work.

Each phase ships with its CLAUDE.md / `docs/` updates per the repo's phase-gate
ritual.

## Risks & open questions

- **Entry-sector model change** is load-bearing for drone ingress — must keep the
  `getEntrySectors()` API stable and test ≥1 entry/region.
- **Idle-galaxy CPU** at 56 bots across 25 rooms — validated by the math above but
  must be bench-confirmed before the scaling PR merges.
- **Map readability** with icons + counts + tint at 20-25 hexes on a phone — start
  tiny, iterate; counts may need a zoom threshold to avoid clutter in `overlay`.
- **`GalaxyMapLayer` render-tree refactor** (flat `hexLayer` → per-territory
  scalable sub-containers) is the biggest single piece — it touches hit-testing
  (`hitTest` reads hex centres through `clusterRoot.scale`) and the pan/zoom camera.
  Keep the territory grouping + centroid + BFS in the pure `galaxyTerritories.ts`
  helper and unit-test it; verify `hitTest`/pan-zoom still resolve through the new
  nesting. This is why Phase 4 splits 4a (territory + shrink) from 4b (icons).
- **Touch has no true hover** — the shrink triggers on the pressed/selected/current
  territory on touch; desktop uses pointer hover. Documented behaviour, not a fake.
- **Faction contiguity is a generator invariant** — if the generator ever emits a
