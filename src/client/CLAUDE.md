# CLAUDE.md ‚Äî src/client (Eyes and Ears)

`src/client` is everything the player sees and hears. It is **not** the source of truth ‚Äî the server is. The client's job is to make latency invisible through prediction (Phase 3) and ghost projectiles (Phase 4), while deferring to server reality whenever the two disagree.

Read the root [CLAUDE.md](../../CLAUDE.md) for project-wide invariants before editing.

---

## Forbidden Imports (CI-enforced)

Never import from `src/client/`:

- Server-only networking: `colyseus` (the *server* Colyseus package; use `colyseus.js`)
- `@colyseus/ws-transport`
- Persistence / Node-only APIs: `better-sqlite3`, `express`, `pino`, `worker_threads`, `node:worker_threads`, `fs`, `node:fs`, `http`, `node:http`
- Anything under `src/server/**`

Allowed: `colyseus.js`, `react`, `react-dom`, `@mui/material`, `@emotion/*`, `pixi.js` v8, `pixi-viewport`, `howler`, `zustand`, `src/core`, `src/shared-types`.

## Zustand Purity (invariant #2, strictly enforced)

**No spatial fields may live in the Zustand store.** This is the single most important client-side rule.

- Forbidden keys (lint-blocked in `src/client/state/store.ts`): `x`, `y`, `vx`, `vy`, `angle`, `rotation`, `position`, `velocity`.
- What Zustand *is* for: `connectionStatus`, `sectorName`, `hullPct`, `ammo`, `sectorAlert`, `playerId`, dev-overlay toggles, HUD flags.
- What Zustand is *not* for: anything that updates every frame.

Why: Zustand triggers React re-renders on subscription changes. Putting per-frame spatial data in Zustand would cause 60 Hz React re-renders, which is a performance catastrophe. Spatial state lives in a plain-object render mirror that Pixi polls directly ‚Äî no React involvement.

## Renderer worker boundary (2026-05-14, Phase 3 of OffscreenCanvas migration)

The renderer is migrating off the main thread onto a dedicated Web Worker via `OffscreenCanvas`. Phase 3 has landed the scaffolding; Phases 4 + 5 host the actual Pixi `Application`s in the worker. See [plan](../../../.claude/plans/humble-strolling-coral.md) for the full multi-phase roadmap.

- **`IRenderer.getFeedback(): RendererFeedback`** is the closed-set channel for renderer ‚Üí main-thread data each frame. Today: in-renderer field, populated at the tail of `update()`. Future (Phase 4): main-thread cache, populated by `FEEDBACK` postMessages from the worker. **Adding a new field requires a phase-gate review** ‚Äî every entry expands the per-frame postMessage payload. Phase-gated additions to date: `liveBeamRenderedFromX/Y` (smoke handoff 2026-06-06, Issue 1) ‚Äî the ACTUAL drawn live-beam origin (the `BeamSpritePool` sprite transform), published as `data-beam-rendered-from-x/y`. It exists because `data-beam-from-x/y` RECOMPUTES the origin from the live ship pose and so passes even when the drawn beam is frozen; the render-cache detach bug (Bug #1) is only observable by reading what's actually drawn. `placementScreenX/Y` (same handoff, Issue 5) ‚Äî the screen projection of the structure placement ghost pose, published as `data-placement-screen-x/y`, so the world-anchored confirm can sit over the blueprint. `placementConfirmSeq` (WS-10 R2.5) ‚Äî a MONOTONIC counter the renderer bumps on a DESKTOP mouse left-click during placement; `gameRafLoop` edge-detects a change and commits the placement at the chosen point via `commitChosenPlacement`, then clears `placementKind`. A monotonic seq (not a one-shot bool) is robust to the FEEDBACK cache being read across several RAFs ‚Äî the value only moves on a real click. `hoveredPickId` (WS-10 R2.4) ‚Äî the id of the entity the desktop pointer is hovering over (set from pointer-move via `pickEntityAt`), driving the lighter `HoverBracket` outline; published as `data-hover-pick-id`. Renderer-local ‚Äî NEVER Zustand (it updates at pointer-move cadence; #2). When you add a `RendererFeedback` field, thread it through all three worker sites (`renderer.worker.ts` FEEDBACK post, `WorkerRendererClient.emptyFeedback` + the FEEDBACK handler) or the worker render path silently lacks it.

- **The wire format is [`render/worker/protocol.ts`](render/worker/protocol.ts).** Discriminated unions in both directions; structured-cloneable variants only. No DOM handles, no Pixi handles, no functions. Locked by `render/worker/protocol.test.ts` (structuredClone roundtrip + compile-time exhaustiveness).

- **Forbidden imports inside `src/client/render/worker/**`** (CI-enforced): `react`, `react-dom`, `@mui/*`, `@emotion/*`, `zustand` ‚Äî the worker has no DOM. State crosses the boundary via the protocol; main thread reads Zustand and posts the resulting messages.

- **`RenderMirror` is the per-frame wire payload** ‚Äî already plain data (no Pixi handles, no functions). Any new field on `RenderMirror` must remain structured-cloneable.

- **pixi-viewport does NOT work inside an OffscreenCanvas-hosted runtime** ‚Äî its Drag plugin calls `addEventListener` on `events.domElement`, which is undefined in a worker. Spike-verified 2026-05-14 (commit `2dd11d3`). Replacement: hand-rolled `Camera` class in the worker, prototype at [render/__offscreen-spike__/spike-worker.ts](__offscreen-spike__/spike-worker.ts).

- **Pointer / wheel / touch events are forwarded from the main thread.** The main thread installs DOM listeners on the gameplay canvas, serialises each event via the `SerialisedPointerEvent` / `SerialisedWheelEvent` shapes, and posts as `POINTER_EVENT` / `WHEEL_EVENT`. The worker's `Camera` consumes via a `onPointerDown / Move / Up` state machine.

## Sizing default: start tiny, grow on request

When you build any new UI element ‚Äî modal, panel, card, button, chip ‚Äî **default to as small as humanly possible**. Especially on mobile. We will iterate it bigger when the user reports it's hard to hit or read; the reverse direction (shipping huge and shrinking back down) wastes review cycles and feels worse.

Concretely:
- Card silhouettes / icons: start ~24‚Äì36 px. Don't reach for 64 px without a reason.
- Body text in HUD / overlays / cards: 9‚Äì11 px. Captions 8‚Äì9 px. Reserve 12+ px for primary CTAs.
- Dialog `maxWidth`: prefer `xs`; only step up to `sm` if content genuinely needs it.
- Padding inside cards / dialogs: `p: 0.5`‚Äì`0.75` (4‚Äì6 px). Avoid MUI defaults (which are 16‚Äì24 px).
- Touch targets that absolutely must be 44 px (iOS HIG) ‚Äî only fight for that on PRIMARY confirm actions, not every chip.
- Buttons: `size="small"` by default, with `fontSize: 11`.

The bias: it's easier to hear "this is too small, bump it 10 %" than to undo the visual heaviness of a default-sized MUI component cluster. The 2026-05-12 ship-picker rebuild is the canonical example ‚Äî `maxWidth="sm"` + full kind descriptions + 64 px silhouettes + 22 px stat chips read as "absolutely huge" in playtest; the rebuild dropped to `xs`, 36 px silhouettes, a one-line stat readout, and 11 px buttons.

## Layout Slot System (2026-05-10)

In-game React overlays are positioned via **named slot anchors**, never hand-placed. The system lives in [layout/](layout/) ‚Äî see [docs/features/mobile-layout.md](../../docs/features/mobile-layout.md) for the full story.

- **Anchors are defined in [layout/anchors.ts](layout/anchors.ts).** A 3√ó3 grid (`top-left` ‚Ä¶ `bottom-right`) plus two specials (`fullscreen`, `transit`). Adding an anchor is fine; mutating an existing anchor's CSS is a code-review prompt because every widget already depends on it.
- **`--mobile-edge-inset` CSS var** ([index.html](index.html)): default `16px`, bumped to `40px` by `(orientation: landscape) and (pointer: coarse)`. The `bottom-left` and `bottom-right` anchor hosts use it so joystick / fire / boost sit further from the bezel when a phone is held sideways. Other anchors keep `16px` everywhere ‚Äî they're not thumb-territory.
- **Widgets render into a slot via `<Slot anchor="...">`** ([layout/Slot.tsx](layout/Slot.tsx)). Slots `createPortal` into the matching host element registered by `<LayoutProvider>`. **No widget sets its own `position`, `top/left/right/bottom`, `zIndex`, or safe-area insets** ‚Äî those are owned by the anchor host. ## Client Prediction + Ghost Projectiles

- **Prediction** (Phase 3): the client predicts its own ship by stepping the same `src/core/physics/World` the server uses, then reconciles against authoritative snapshots. Drift ‚â• `LERP_THRESHOLD` (0.05 u position / 0.001 rad angle, just above float32 noise) triggers a visual lerp; lerp duration scales with magnitude (3‚Äì18 frames). Remote ships interpolated with a 100 ms display-delay buffer.
- **Drones are PURE snapshot-interpolated ‚Äî no client brain (2026-05-18, drone-snapshot-interpolation pivot; supersedes the chapter-2 re-sim/lockstep design).** The client runs **no** drone AI. Drone pose comes off the binary swarm wire and is rendered via `interpolateSwarmPose` (display-delay buffer + teleport guard), exactly like asteroids/remote players. `ColyseusClient.updateMirror` writes that interpolated pose into the swarm mirror entry **and kinematically into the predWorld drone body** each frame, so render == collision and the local ship still bumps drones; the server stays 100 % hit-authoritative (there is no client drone ray). **ONE-POSE-PER-FRAME RULE (load-bearing, enforced 2026-05-19 ‚Äî was previously only *claimed* here):** `interpolateSwarmPose` is called for a given drone **exactly once per frame ‚Äî in `updateMirror`, at that frame's single `now`**. Every other consumer (the `PixiRenderer` sprite for `kind===1`, the predWorld collision body, the turret/laser aim `buildLocalAimTargets`, health bars, labels) **MUST read the resolved `entry.x/y/angle` and MUST NOT call `interpolateSwarmPose` again.** The greppable, unit-lockable seam for this is [`resolveEntityDisplayPose`](net/swarmDisplayPose.ts) ‚Äî use it; never re-interpolate a drone in a consumer. Re-interpolating in a consumer resolves the pose at a *different* `now` than the frame's single resolution (render-now / tickPhysics-now ‚âÝ updateMirror-now, raf-jitter-amplified, a whole frame under the 30 Hz worker sprite gate), so the sprite, collision body and beam disagree every frame: that **was** the on-device "drones jitter like two things fighting / laser jitters vs where it's drawn" bug (2026-05-19, capture `‚Ä¶-jfagww`; `0e24448`'s aim-via-`interpolateSwarmPose` was directionally right but added a 3rd divergent-`now` site ‚Äî subsumed). The accepted residual is a smooth ‚â§1-frame aim lead-lag (`buildLocalAimTargets` runs in `tickPhysics`, before `updateMirror`, so it reads the prior frame's pose), never per-frame jitter. **Exception:** asteroids (`kind===0`) keep render-now `interpolateSwarmPose` off the poseRing ‚Äî they are locked/static server-side, were never the jitter complaint, and `syncSwarmIntoPredWorld` still poses their bodies from the raw decoded `entry.x/y` (the decoder is deliberately UNCHANGED ‚Äî it still writes the raw pose + feeds the poseRing; the per-frame resolve overwrites the drone `entry.x/y` before any consumer reads it). `SnapshotMessage.drones[]` is a slim `{ id, mountAngles?, shieldDown? }` turret/shield slice only ‚Äî x/y/vx/vy/angle/angvel flow ONLY on the binary channel. `_aiController` survives as a **hostility ledger only** (`markHostile`/`isEntityHostileToPlayer`, fed by `damage`/`bot_aggro`); its brain is never ticked. **RETIRED:** `tickClientAi`, `AiController.tickOnly`, `partitionDronesByRelevance`/`droneRelevance.ts`/`DRONE_RESIM_BUDGET`, `_droneSnapshotAnchored`, `_droneRenderOffsets`/`_droneLastSnapDist`, `swarm_snap_diagnostics`/`stats.swarmSnap*`, the `Reconciler.reconcile` drone replay-seed (6th param), `anchoredDroneReseedSmoothing`. ## Snapshot transport ‚Äî DataChannel + WebSocket coalescing (2026-05-30, swift-otter Phase 4 iteration 3)

Full architecture story: [docs/architecture/webrtc-datachannel-snapshot-transport.md](../../docs/architecture/webrtc-datachannel-snapshot-transport.md). Three load-bearing rules below ‚Äî if you touch any of these surfaces, all three apply.

- **The DataChannel is DEFAULT-ON for real users (flipped 2026-06-06), DEFAULT-OFF under automation.** The opt-in `?webrtc=1` was flipped to default-on after on-device evidence that the DC establishes over LAN host-candidate ICE and carries ~99% of snapshots over UDP (avoiding the WS/TCP head-of-line amplification that stretches a brief WiFi-radio stall into a ~530 ms snapshot gap ‚Äî the on-device "jumps"). Reliable WS stays as automatic fallback when the DC can't establish (symmetric NAT / cellular without TURN), so default-on is safe. The decision is the pure [`net/webrtcEnable.ts`](net/webrtcEnable.ts) `webrtcEnabledFromSearch(search, isAutomation)`: `?webrtc=0`/`=1` override; otherwise ON unless `navigator.webdriver` (Playwright). **The automation gate is load-bearing:** the per-PR E2E suite must stay WS-deterministic, and the **netgate MUST keep measuring the WS path** ‚Äî the DataChannel is peer-to-peer UDP and BYPASSES the netgate's injected-latency HTTP+WS proxy, so a webdriver default-on would silently invalidate the gate. `webrtc-vs-ws-recv-gap-comparison.spec.ts` passes `?webrtc=1` explicitly to exercise the DC arm. Mirrors the `?diag` webdriver-gate pattern. (Caveat the on-device A/B left open: the ~530 ms radio gaps are intermittent and a clean same-session WS-vs-DC comparison did NOT reproduce them on either transport, so the gap-reduction is theory + one suggestive sample, not repeatably proven ‚Äî the change is a low-risk bet backed by "DC works + WS fallback", not a proven gap-fix.)

- **Snapshot APPLY happens at RAF tick, never inline in `onMessage`.** Both transports converge at the same `snapshotCoalescer` (`src/client/net/SnapshotCoalescer.ts`), which drains in `tickPhysics ‚Üí processPendingSnapshot`. The WS handler `room.onMessage('snapshot', ...)` calls `snapshotCoalescer.enqueue(snap)` and returns; the DC path calls `_receiver.enqueueBinary(rawBytes)` and returns. Inline apply via `?coalesce=0` is the legacy A/B mode ‚Äî production gameplay must never run that way (it serialises decode + reconcile inside the message dispatch frame and produces the loaf class iteration 3 was built to remove).

- **Byte-level coalescer on the DC path.** `dc.message` listener in `DataChannelTransport` MUST call `_receiver.enqueueBinary(buf)` (stores latest raw bytes, no decode), NEVER `_receiver.handleBinary(buf)` (decode + dispatch). `_receiver.drain()` is called once per RAF from `tickPhysics`. The reason: Pattern B network bursts deliver N queued snapshot frames into the JS event loop in a single frame; decoding all N synchronously produced 235 ms `dataChannelTransport.ts` loafs. SCTP `ordered: true` (the default) guarantees no reordering, so "latest enqueued = freshest" is sound. `handleBinary` is preserved as the direct-dispatch path for unit / integration tests; production code never calls it. Lock: 6 cases in [`net/dataChannelTransport.test.ts`](net/dataChannelTransport.test.ts) under the "raw-bytes coalescer" describe.

- **`intervalMs` MUST be wire-arrival time, not RAF apply time.** [`snapshotPerfStats.ts`](net/snapshotPerfStats.ts) `applySnapshotPerfStats` takes a `wireArrivalAtMs` arg (from `_lastSnapshotRecvAtMs`, the snapshot's recv time set inside `logSnapshotRecvTelemetry`). `intervalMs = wireArrivalAtMs - lastSnapshotAt` (also wire time of previous apply). Why: the snapshot coalescer makes apply cadence RAF-bound (~16‚Äì33 ms); the downstream [`rttLookaheadUpdater.ts`](net/rttLookaheadUpdater.ts) REJECTS RTT samples outside the 35‚Äì75 ms `STEADY_STATE_INTERVAL_*` band ‚Äî apply-bound samples saturated the rejection filter, Welford starved, `leadTicks` inflated, `ticksAhead` regressed. Wire cadence is ~50 ms at 20 Hz ‚Üí in-band ‚Üí healthy. **The rule generalises: anywhere we use APPLY timing to estimate a NETWORK characteristic, that's a latent bug ‚Äî use the wire-arrival timestamp.** Lock: [`tests/unit/snapshotPerfStats.intervalMs.test.ts`](../../tests/unit/snapshotPerfStats.intervalMs.test.ts) (3 cases).

- **`onStateChange` (Colyseus schema diff) uses HYBRID inline-first / defer-rest.** First `onStateChange` per RAF window runs `syncMirror` inline (preserves reconciler drift baseline behaviour); subsequent within same window go through `_pendingStateForSync`, drained at top of next `tickPhysics`. Cap of 2 syncMirror calls per RAF (one inline + one drain). Full-defer (always defer, never inline) was tried and reverted: caused `maxDriftUnits` regression in 5-rep netgate median (12 ‚Üí 36) ‚Äî mechanism specific to the netgate latency proxy's per-byte TCP jitter pattern, not reproducible under CDP-emulated jitter, so we cannot localise without proxy-level instrumentation. **General pattern for backpressure-tolerant work inside Colyseus message handlers**: keep the first call per frame synchronous (existing assumptions about callback ordering hold); defer only redundant work the burst would otherwise duplicate.

## Input Throttling Discipline (2026-05-06)

The client may suppress redundant input sends ONLY when both the current and previously-sent input states are **fully idle** (every control bit false). Any held key ‚Äî thrust, turn, boost ‚Äî must be re-sent every tick, with an additional 250 ms heartbeat in idle to keep the server's session alive.

Why narrowed to all-idle: when a held input has been throttled, the server's worker re-applies the held state each tick under its synthesised-ack contract (see [src/core/CLAUDE.md](../core/CLAUDE.md) ‚Üí Input Queue Contract). When the client THEN sends a state change at a tick higher than the synthesised ack, the worker's max-tick-clamp jumps the ack past the intermediate ticks ‚Äî silently skipping a physics step that the client's local prediction DID apply. On a fast-moving ship this surfaces as a ~8 unit drift per state-change event, with `corr` rate sticking around 20‚Äì30 %. Restricting throttling to all-idle frames is safe because held all-idle adds zero impulse ‚Äî the server skipping a tick is physically equivalent.

**Rule:** any future per-tick stream that adds throttling needs the same audit: when the held state is "active" (changes physics), the server's queue must stay populated. See `docs/LESSONS.md` 2026-05-06 follow-up for the full incident.

## Multi-mount mirror surfaces (Phase 2c‚Äì4c, 2026-05-11)

The renderer mirror exposes per-mount data on three surfaces, all keyed by mount id (from the ship-kind catalogue):

### Beam length is `scale.x √∑ texture.width`, NOT `scale.x = worldLen` (Equinox laser issue, 2026-06-14)

`BeamSpritePool` draws each beam segment as a stretched `Sprite`. A Pixi sprite renders `worldLen = scale.x √ó texture.width`, so the world length must be set as **`scale.x = worldLen / texture.width`**. The solid core uses the 1√ó1 `Texture.WHITE` (√∑1, fine), but the **gradient + fade textures are 256 px WIDE** (they have to be ‚Äî that's where the horizontal fade lives). Setting `scale.x = worldLen` on those rendered the fade tail **256√ó too long** (a 75 u tail ‚Üí ~19 200 u): the "fade" stretched off any screen at any zoom, so in empty space the beam looked SOLID to infinity and "never tapered" ‚Äî the long-standing "laser renders infinitely / flies off the screen" bug. **Headless unit tests fall back to `Texture.WHITE` (1√ó1) for the canvas textures, so they could NEVER see this** ‚Äî the fix had "passed" four rounds of screenshots while staying broken on-device. ## Damage numbers and health bars

- `mirror.pendingDamageNumbers` and `mirror.pendingHealthBarHits` are per-frame **drain queues** populated by `ColyseusClient.handleDamage()` and consumed by `PixiRenderer.update()`. They are arrays, not maps ‚Äî every entry is consumed once per frame.
- `DamageNumberManager` ([render/DamageNumbers.ts](render/DamageNumbers.ts)) ‚Äî **accumulator model (2026-05-30, plan melodic-engelbart Step 4).** One floating-number bucket per `targetId` at any time. Each `spawn(targetId, x, y, damage, tag?)` either adds to the existing bucket (running total grows, font scales via `fontScaleForTotal`, stay-window resets, text re-anchors at the new hit coord) OR opens a fresh bucket (evicting the bucket with least life remaining if at `POOL_CAP=20`). After `STAY_FRAMES` (~1 s @ 60 Hz) with no new hits the bucket fades over `FADE_FRAMES` (~0.5 s) and is destroyed. Pre-accumulator the manager `new Text(...)`-ed per hit at ~30 Hz under held fire ‚Äî aggregating into ~1 bucket per active target reduces text-construction churn by an order of magnitude (the dominant non-FX allocation cost surfaced by the imperative-taco hostile CDP profile). `cancelByTag(tag)` subtracts the predicted contribution from each bucket the tag touched and destroys buckets whose total drops to zero; per-tag contributions are tracked in a lazy `Map` allocated only on tagged adds (auth-only buckets ‚Äî the common case ‚Äî never allocate it). Server provides `hitX`/`hitY` in `DamageEvent`; falls back to entity-pose if absent.
- `HealthBarManager` ([render/HealthBars.ts](render/HealthBars.ts)) shows a bar above an entity only when the local player has just hit it (`evt.shooterId === localId`). Bar fades after 2 s with no consecutive hits and removes after 2.5 s. Position is read from `mirror.ships` or `mirror.swarm` each frame so the bar tracks moving targets.

## Phase A3 ‚Äî Renderer decision logic extraction (2026-05-13)

Per-entity sprite-update decisions (create / rebuild / reposition / skip) live in `src/client/render/spriteUpdateDecisions.ts` as pure functions, NOT inlined inside `PixiRenderer.ts`. Don't inline new decision branches in `PixiRenderer.ts`.

Current functions:
